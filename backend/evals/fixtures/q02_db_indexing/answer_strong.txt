An index is a separate data structure, usually a B-tree, that maps column values to row locations. Instead of a full table scan, the database walks the tree in logarithmic time to find matching rows. The trade-off is that it isn't free: every insert, update, and delete now has to maintain the index too, and it consumes storage. So adding one hurts in a few cases. On a write-heavy table, each extra index slows down every write. On a low-selectivity column — say a boolean or a status field with three values — the index barely narrows anything, and the planner will often ignore it and scan anyway, so you pay the write cost for nothing. There's also the case where an index misleads the planner into a worse plan. One thing worth knowing is covering indexes: if the index contains every column the query needs, the database can answer from the index alone without touching the table, which is a big win for hot read paths.
