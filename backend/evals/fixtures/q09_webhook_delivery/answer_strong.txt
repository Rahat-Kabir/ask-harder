The producing side should never make the HTTP call inline — it writes an event row or queue message in the same transaction as the state change, ideally via an outbox pattern so an event is recorded if and only if the action committed. Delivery workers consume from that durable queue and POST to the customer endpoint with a timeout.

Failures are the normal case, not the exception: customer endpoints are down, slow, or buggy constantly. Retries use exponential backoff with jitter — say from thirty seconds up to hours, over a day or so. After the retry budget is exhausted, the event goes to a dead-letter store the customer can inspect and replay from a dashboard, and an endpoint that fails everything for days gets auto-disabled with a notification, so we stop burning capacity on it.

This design is at-least-once: a worker can deliver, crash before acking, and deliver again. So every event carries a stable event ID for consumer-side deduplication, and the payload is signed with an HMAC and timestamp so the consumer can verify it came from us. Two things I'd call out explicitly: per-endpoint queues or concurrency limits, because one customer's 30-second-timeout endpoint must not stall deliveries for everyone else sharing the workers; and ordering — I'd document delivery as unordered and put a sequence number in the payload, because guaranteeing order across retries means head-of-line blocking on every failure.
