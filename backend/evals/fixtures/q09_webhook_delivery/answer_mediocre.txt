I wouldn't send the webhook directly from the code that creates the event — I'd write the event to a queue and have worker processes consume it and make the HTTP calls. If the customer endpoint returns an error or times out, the worker retries a few times with exponential backoff so a brief outage on their side doesn't lose the event. After some number of failures, you give up and maybe log it somewhere. The queue also means a burst of events doesn't overwhelm the senders.
