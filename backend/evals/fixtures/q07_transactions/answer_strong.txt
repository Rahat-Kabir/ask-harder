Isolation levels exist because concurrent transactions interleave, and naive interleaving produces anomalies: dirty reads, where you see another transaction's uncommitted write; non-repeatable reads, where the same row gives two different values within one transaction because someone committed in between; and phantoms, where a repeated range query returns new rows. The SQL standard defines four levels that progressively forbid these: read uncommitted allows everything, read committed stops dirty reads, repeatable read stops non-repeatable reads, and serializable makes the outcome equivalent to running the transactions one at a time.

The trade-off is throughput: stronger isolation means more locking or more aborted-and-retried transactions, so you pay in concurrency. That's why nobody defaults to serializable — Postgres defaults to read committed, and you raise the level only for the transactions that need it.

In practice modern databases complicate the textbook picture because they use MVCC: readers get a consistent snapshot instead of taking locks, so readers never block writers. Postgres's repeatable read is actually snapshot isolation, which prevents the standard anomalies but still permits write skew — two transactions each reading the same data and writing disjoint rows can together violate an invariant. Its serializable level detects those dangerous patterns and aborts one transaction, so your application needs retry logic.
