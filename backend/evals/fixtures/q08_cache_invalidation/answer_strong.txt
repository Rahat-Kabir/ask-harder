First question is how stale is acceptable, because that decides the strategy. The baseline is a TTL: bounded staleness, dead simple, works even when you don't control the writes. If writes flow through your own system, you layer explicit invalidation on top — on write, either update the cache (write-through) or delete the key and let the next read repopulate it. Deleting is usually safer than updating in place because it can't write a stale value over a fresh one. A trick I like for entities with a version or updated_at field is putting the version in the cache key itself — invalidation becomes automatic because new data is simply a different key, and you never race on overwrites; the old keys just age out.

The big failure mode you've introduced is the stampede: a hot key expires, and suddenly a thousand concurrent requests all miss and hammer the slow service at once — the exact thing the cache was protecting against, except now it arrives as a synchronized spike. Mitigations: per-key locking or request coalescing so only one caller regenerates while the rest wait or serve the stale value, jitter on TTLs so keys don't expire in waves, and ideally serve-stale-while-revalidate for hot keys. The other new failure mode is the cache becoming a hard dependency — if Redis dies and your origin can't take the full unshielded load, the cache outage becomes a total outage. You want load shedding or a degraded mode planned for that, not discovered during it.
