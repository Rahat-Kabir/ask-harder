The naive approach is OFFSET/LIMIT, and it breaks in two distinct ways at this scale. First, performance: OFFSET doesn't seek, it counts — the database has to produce and discard every skipped row, so page one is instant and page ten thousand does a half-million-row scan. Second, correctness: the dataset moves underneath the client. If rows are inserted or deleted between page requests, offsets shift, and the client silently sees duplicates or skips items — which is poison for anything like data export or sync.

So I'd use keyset pagination. Pick a stable, indexed sort order — say created_at descending — and have each page request continue from the last row of the previous page: WHERE (created_at, id) < (last_created_at, last_id) ORDER BY created_at DESC, id DESC LIMIT 100. The id in there matters: created_at can have duplicates, so you need a unique tie-breaker column or rows on the boundary get skipped. With a composite index on (created_at, id) every page is an index seek — constant time at any depth, and inserts can't cause duplicates because the position is anchored to a row value, not a count.

For the API surface, I'd encode the position into an opaque cursor token — base64 of the sort values — rather than exposing the raw columns. That keeps clients from constructing cursors by hand and lets us change the underlying sort without breaking them. The response carries next_cursor; no cursor means first page. Trade-off to state honestly: you lose "jump to page 47", which keyset can't do — almost always fine for APIs, since consumers iterate rather than jump.
