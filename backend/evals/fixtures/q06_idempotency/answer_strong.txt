The core problem is that the timeout is ambiguous: the server may have completed the operation and the response was lost, so a naive retry duplicates the side effect — a double charge, a double order. The standard fix is an idempotency key. The client generates a unique key per logical operation, usually a UUID in a header, and sends the same key on every retry of that operation. On the server, the first request stores the key along with the response it produced; any later request with the same key skips execution and returns that stored response, so the client can't tell the difference between the first attempt and a replay — which is exactly what you want.

The subtle part is making the key check atomic. If you check-then-insert in two steps, two concurrent retries can both pass the check and both execute. So the key goes in a table with a unique constraint, inserted in the same transaction as the side effect — the second writer hits the constraint violation and either waits for the first to finish or returns its result. You also have to handle the in-flight case: same key arrives while the first attempt is still executing, which should return 409 or block, not re-execute. Keys should be scoped per client and expired after some window, say 24 hours, so the table doesn't grow forever and key reuse by accident doesn't return stale results.
