I'd start with a token bucket per API key: each bucket has a capacity and a refill rate, every request takes a token, and an empty bucket means 429 with a Retry-After header. I prefer token bucket over a fixed window because it permits short bursts without allowing a sustained overage, and it avoids the boundary problem where a fixed window lets you double the limit by straddling the reset.

The interesting part is where the state lives. With a single node you could hold buckets in memory, but a public API runs behind a load balancer with many instances, so per-node counters undercount globally. I'd keep the buckets in a shared Redis, with the check-and-decrement done atomically in a Lua script to avoid races between instances. If the latency of a Redis round trip per request is too much, a common compromise is local pre-aggregation — each node keeps a small local allowance and syncs with Redis periodically — trading a little accuracy for a lot less coordination.

Finally, failure behavior: if Redis goes down, I'd fail open and let requests through while alerting, because for most public APIs rejecting all traffic is worse than briefly losing rate enforcement. For an expensive or abuse-prone endpoint you might choose fail-closed instead. That choice should be explicit, not accidental.
